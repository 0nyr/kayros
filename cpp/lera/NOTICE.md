# NOTICE — vendored Lera-Romero BPC code (`cpp/lera/`)

This directory vendors the branch-price-and-cut (BPC) solver for the TDVRPTW originally written by **Gonzalo Lera Romero** (Grupo de Optimización Combinatoria, Departamento de Computación, Universidad de Buenos Aires), together with its supporting `goc` library, as adapted in Florian Rascoussier's `TDVRPTW-solver` repository.

The algorithm is described in: Lera-Romero, Miranda-Bront, Soulignac, *Dynamic programming for the time-dependent traveling salesman problem with time windows* (and the companion TDVRPTW BPC line of work published from the same codebase, Networks 2019 / INFORMS JOC).

## Local modifications (kayros)

Kept deliberately minimal; the reference for algorithmic behavior is Lera-Romero's `8329844` plus Onyr's half-port intent:

1. `goc/include/goc/goc.h`: re-added the two `goc/vrp/*` includes (match `8329844`).
2. `main/include/bcp/bcp.h` + `main/src/bcp/bcp.cpp`: the half-ported `nyr::VrpSolutionRecord&` parameter (a type that no longer exists) was replaced by (a) restoring Lera's original `Run(goc::VRPSolution*)` signature, and (b) a new optional `on_incumbent` callback member invoked on every new upper bound (anytime surfacing, plan 2 Stream 5 M5.0/M5.2).
3. Build: this subtree is compiled by kayros' own CMake (`KAYROS_WITH_LERA=ON`), never by upstream's CMakeLists.
4. **HiGHS backend (M5.1, kayros-added)**: `goc/{include,src}/goc/linear_programming/highs/` — `MatrixFormulation` (solver-agnostic in-memory `Formulation`) + `highs::solve_lp`/`solve_bc` mirroring the cplex pair, stateless materialization per solve (duals postsolved by HiGHS; no presolve weakening needed; single-threaded per kayros' one-run-one-thread convention). Compile-time switch `LERA_LP_BACKEND=cplex|highs` in `lp_solver.cpp`/`bc_solver.cpp` (`GOC_LP_BACKEND_HIGHS`); unused cplex includes dropped from `cg_solver.cpp`; `variable.h` gained `friend class MatrixFormulation`. Unsupported in the HiGHS `solve_bc` (fail loudly, unused by the BPC path): MIP starts, branch priorities, cut callbacks, lazy constraints. Known perf gap vs CPLEX: no simplex-basis warm start across CG iterations (rebuild per solve) — revisit at M5.2+.
5. `main/src/labeling/bidirectional_labeling.cpp`: `RouteDuration` → `goc::Route` conversion at the solution-pool exit; `nyr/include/nyr/vrp/delta.h`: explicit `SubpathHash` (GCC 15).
6. **Stopping / anytime compliance (M5.2, kayros-added)**: new `goc/include/goc/time/deadline.h` (`goc::Deadline`, absolute wall-clock deadline). `BCP` derives one deadline from `time_limit` at `Run()` and every component takes its residual budget from it instead of nested stopwatch re-derivations: CG (`ProcessNode`), strong-branching LPs (`EstimateBound`, which previously ran **unlimited** — `solve_colgen` reset the shared `LPSolver.time_limit` to `Duration::Max()`), the freeze heuristic (previously got the **full** TL as residual), and the root cut loop. New interruption points: `SeparateCuts` O(n³) enumeration, `IterativeMerge`/`LastArcMerge` in the bidirectional labeling, and the labeling's solution-pool repricing (post-deadline tail cut). Correctness guards: an aborted strong-branching probe or truncated pricing pass stamps `TimeLimitReached` so a deadline hit can never masquerade as an optimality proof (including `LastArcMerge` truncation no longer reporting `Finished`, and the `z_lb = z_ub` gap-close now conditional on a fully explored tree); the freeze heuristic accepts a TL-truncated MILP incumbent when one exists. `solve_colgen` now takes the CG optimum from the last in-loop LP solve instead of an unconditional re-solve (which wasted a solve and leaked `Duration::Max()` into later LPs).
7. **Warm start (M5.3, kayros-added)**: `BCP::SetInitialIncumbent(double)` — public setter for an initial UB known by the caller (the value of warm-start columns the bridge adds via the pre-existing `SPF::AddRoute` injection point). The bound must be in the master's own arithmetic (checker-exact since item 8; a bound tighter by dust than the master's costs could prune the true optimum) and no incumbent callback fires for it.
8. **Checker-exact column costs (M5.6 stage A, kayros-added, bridge-side)**: every route entering the SPF (seeds, warm-start columns, priced columns) is repriced by the kayros checker-exact fold (`cpp/core/route_eval` over the payload's raw MAMUT ATFs — Lera's preprocessing folds service/waiting into travel times, so the fold runs on the verbatim data), making all master objective coefficients, LP bounds, UBs and the reported value checker-exact; the final value is re-summed in the checker's canonical route order (bit-identical to `compute_solution_cost`). Lera's `BestDurationRoute` remains the *search* arithmetic (pricing reduced costs, labeling); safety: pricing emits only reduced costs < −1e-6 (goc epsilon) while the Lera-vs-checker cost dust is ~1e-12, so priced columns stay strictly improving under checker costs, and a checker-infeasible priced column fails loudly (certification integrity). **Pricing completeness is still modulo Lera's epsilon arithmetic** (a checker-optimal route could in principle be dominated away by dust inside labeling) — the stage-B labeling swap that would remove this residue was **deliberately skipped** (Onyr, 2026-07-06 night): values are checker-exact, the checker repricing referees every result, and the residue is a dust-probability with zero divergences observed across all gates. Certificates must keep wording it honestly.

9. **Stepwise-ATF (Rifki-style) support (M5.7, kayros-added)**: `main/src/labeling/bidirectional_labeling.cpp` gained `continuize_value_jumps` applied to each reverse arrival function in `reverse_instance`. Stepwise travel times produce interior *plateaus* in the forward arrival function (a tau down-step encoded as a slope −1 arrival segment); `dep = arr.Inverse()` turns each plateau into a value jump, the reflected reverse arrival inherits it, and inverting *that* leaves an interior domain gap where `DepartureTime`/`Value` fails (Lera's original padding only handles the boundary waiting plateau — IGP instances never produce interior ones). The helper bridges each upward value jump with a steep segment of width ≤ 1e-3 that stays *below* the following piece, so reverse arrivals are never over-estimated: bounds remain valid and no wrong optimum can be certified; the measure-delta under-approximation sliver is the same documented dust class as the epsilon-dominance residue (item 8). The forward-direction counterpart lives bridge-side in `kayros.lera._continuize_breakpoints` (duplicate-x breakpoints — tau up-steps — mollified before Lera ever sees them, matching the checker's left-continuous evaluation at the jump point exactly). Two follow-on dust guards in the labeling (same M5.7 scope): a label's `rw` and `dom(duration)` can disagree by mollifier dust (≤ 1e-3, beyond goc EPS), so boundary evaluations of label duration functions are clamped into the function's domain (`duration_at` in `bidirectional_labeling.cpp::Merge` and `monodirectional_labeling.cpp` alpha/beta) — Lera search arithmetic only, never certification arithmetic; and the bridge's stage-A guard skips Lera's empty-path INFTY pool placeholder (`AddSolution`'s `Route({}, 0, INFTY)`), which is not a column and pre-M5.6 entered the SPF as a harmless artificial-like column. **Refuted 2026-07-08:** the "no wrong optimum can be certified" claim in this item is false. Counterexamples: independent heuristic campaigns produced checker-valid solutions strictly below 43 of the 160 Rifki2020 certified optima (up to 1.46%), and certification is warm-start-dependent (TDVRPTW n=20 Rifki-16 certifies `Optimum` 8485 / 8376 / 8361 from stored-BKS / cold / counterexample warm starts, each in seconds, all three solutions feasible in the solver's own model). The pointwise under-approximation argument does not cover the labeling's *internal* arithmetic on the bridged functions: a 1e-3-wide bridge over a step of height h has slope ≈ 1000·h, so goc's 1e-6 epsilon comparisons and the `duration_at` domain clamps (this item) can misprice a merge by O(h) — enough for `Merge` to discard negative-reduced-cost routes, silently breaking pricing completeness. Do not stamp optimality certificates on stepwise-ATF (jump-carrying) instances until the labeling handles value jumps exactly; all Rifki2020 stamps were retracted from the MAMUT store on 2026-07-08. **Resolved 2026-07-09 (M5.9), by a different fix than "handle value jumps exactly".** The root cause was not the mollifier's approximation. It was pricing incompleteness in goc's numerics: `PWLFunction::Inverse` was built by accumulating per-piece inverses through `Max`, which carried epsilon dust that, compounded over the mollified steep bridges, discarded negative-reduced-cost columns. `Inverse` is now an exact coordinate swap for non-decreasing functions (arrival and departure are non-decreasing), so the mollified continuous path prices completely, and `Value` is left-continuous to match the checker at jump abscissae. With this, the mollifier is retained and sound: the Rifki-16 n=20 reproducer now certifies `Optimum` 8361 cold and warm (was 8376 cold vs 8361 warm), and the fuzzer-discovered Rifki-22 subset reproducer certifies 2496 cold and warm (was 2504 vs 2496). An *exact* value-jump representation (true verticals through the labeling) was implemented and parked: it is currently less sound than the mollified path, because the labeling assumes uniform left-continuity while the time-reversal and composition of a left-continuous step function are not left-continuous. That representation stays as dormant, unit-tested scaffolding (`goc::LinearFunction::is_vertical`/`sweep_endpoints` and the step-capable `Value`/`Inverse`) for a future exact redesign; production never creates a true vertical. Soundness is validated by a randomized differential fuzzer against the ILS heuristic (`tests/test_prover_fuzz_soundness.py`, `tests/td_fuzz.py`): random subsampled instances (value-jump ATFs preserved) must certify no optimum above the ILS feasible cost and no warm-start-dependent value. Re-stamping Rifki2020 remains gated on a large fuzz sweep staying clean plus a nancy.g5k re-certification of Rifki2020 n≤50.

Behavioral changes inherited from the half-port (upstream `TDVRPTW-solver`, not introduced here): subset-row cut violation threshold raised from `> 0` to `> 0.1` in `BCP::SeparateCuts`.
